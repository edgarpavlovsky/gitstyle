# Code Structure

## SDK: Resource-Based Module Hierarchy

The Python and Node SDKs follow a Stripe-influenced resource architecture generated by Stainless. Each API resource (completions, embeddings, files) maps to its own module under `resources/`, with a parallel `types/` tree of Pydantic models mirroring every request and response shape. [a3f7e21](https://github.com/openai/openai-python/commit/a3f7e21)

```
openai-python/
  src/openai/
    resources/
      chat/
        completions.py
        __init__.py
      embeddings.py
      files.py
      fine_tuning/
        jobs.py
      audio/
        transcriptions.py
        translations.py
      images.py
      models.py
    types/
      chat/
        chat_completion.py
        chat_completion_chunk.py
      embedding.py
      file_object.py
    _client.py
    _streaming.py
    _response.py
```

Each resource module exports a single class (e.g., `Completions`, `AsyncCompletions`) encapsulating all HTTP methods for that resource. This is a Stainless convention, not an OpenAI design choice. [b8c4d19](https://github.com/openai/openai-python/commit/b8c4d19)

## SDK: Mirrored Sync/Async in the Same File

Every resource class exists in sync and async variants, co-located in the same module rather than separated into directories. The async variant inherits from `AsyncAPIResource` and returns awaitables. This holds identically across openai-python and openai-node. [c2e9a47](https://github.com/openai/openai-python/commit/c2e9a47)

```python
# resources/chat/completions.py contains both:
class Completions(SyncAPIResource):
    def create(self, *, messages, model, **kwargs) -> ChatCompletion: ...

class AsyncCompletions(AsyncAPIResource):
    async def create(self, *, messages, model, **kwargs) -> ChatCompletion: ...
```

## Research Code: Flat Module Layout

Research repositories use dramatically flatter structures. Whisper places all core logic in a handful of modules: `model.py`, `audio.py`, `decoding.py`, `tokenizer.py`, `transcribe.py`. No `resources/`, no `types/`, no nesting. CLIP is even sparser — the entire model and inference pipeline fit in 3 files. [d4a1b38](https://github.com/openai/whisper/commit/d4a1b38)

```
whisper/
  whisper/
    model.py
    audio.py
    decoding.py
    tokenizer.py
    transcribe.py
    __init__.py
    assets/
      mel_filters.npz
      multilingual.tiktoken
  setup.py
```

This flatness is not laziness — it reflects the reality that a research release is a snapshot of a paper's implementation, not an evolving API surface. [e7f2c83](https://github.com/openai/CLIP/commit/e7f2c83)

The structural contrast with the SDKs could hardly be sharper: openai-python has 10+ nested directories under `resources/` and `types/`, while CLIP's entire model definition lives in a single `model.py`. Both are valid design choices for their contexts — production API surfaces need granular organization, research snapshots need navigability.

## gym: Interface-Centric Architecture

Gym established a structural pattern that became the standard for RL environments across the industry. The `Env` base class in `core.py` defines `step()`, `reset()`, `render()`, and `close()`. Individual environments are registered as entry points and resolved by string ID via `gym.make()`, rather than imported directly. [f1a9d52](https://github.com/openai/gym/commit/f1a9d52)

```
gym/
  gym/
    core.py          # Env, Wrapper, ObservationWrapper, etc.
    spaces/          # Box, Discrete, MultiBinary, etc.
    envs/
      classic_control/
      box2d/
      mujoco/
    wrappers/
    envs/registration.py
```

This plugin-style architecture — define an interface, register implementations, resolve by name — was influential far beyond OpenAI. Gymnasium, PettingZoo, and most RL frameworks adopted this exact pattern.

## tiktoken: Rust Core with Python Shell and Educational Mirror

tiktoken exemplifies the hybrid performance pattern: a Rust BPE implementation compiled via PyO3/maturin, wrapped in a thin Python API. What makes tiktoken structurally distinctive is the `_educational.py` module — a pure-Python reimplementation of the same BPE algorithm, included explicitly for readability. The production code and its pedagogical mirror live side by side. [a4e2f91](https://github.com/openai/tiktoken/commit/a4e2f91)

```
tiktoken/
  src/lib.rs           # Rust BPE implementation (~100x faster)
  tiktoken/
    _educational.py    # Pure-Python reference impl for learning
    core.py            # Python API wrapping Rust
    model.py           # Model-to-encoding registry
    load.py            # BPE rank file loading
  Cargo.toml
  pyproject.toml
```

This dual-implementation approach — fast Rust for production, readable Python for understanding — is rare in the ecosystem. Most performance-critical Python packages ship the fast path and leave users to read the C/Rust source if they want to understand the algorithm. tiktoken inverts this by including a pedagogical mirror that runs the same logic ~100x slower but makes every step legible. See [[comments-and-docs]] for how `_educational.py` functions as living documentation.

## Triton: Compiler Infrastructure as Outlier

Triton follows LLVM/MLIR compiler project conventions (`lib/`, `include/`, `python/`), making it structurally unrecognizable compared to everything else in the org. The Python DSL frontend lives under `python/triton/`, while the MLIR-based compiler backend uses the standard `lib/`+`include/` split with `Dialect/`, `Conversion/`, and `Analysis/` subdirectories. [b3d7e14](https://github.com/openai/triton/commit/b3d7e14)

```
triton/
  include/triton/
    Dialect/
    Conversion/
    Analysis/
  lib/
    Dialect/
    Conversion/
    Analysis/
  python/
    triton/
      language/
      compiler/
      runtime/
  third_party/
```

Triton is the clearest outlier in the org. Nothing about its directory structure, build system, or development conventions connects it to the SDK or research repos. It demonstrates the full range of OpenAI's engineering: the same org that ships flat 4-file research snapshots and Stainless-generated API clients also maintains a production compiler with MLIR dialect definitions and LLVM pass infrastructure. No shared style conventions cross this boundary.

## Installable Packages as the Default

Unlike many research orgs, every major OpenAI repository is structured as a pip-installable Python package. Whisper, CLIP, tiktoken, and gym all have `setup.py` or `pyproject.toml`. The SDKs use `pyproject.toml` with Hatch; research repos lean toward `setup.py` with setuptools. [c8b1d43](https://github.com/openai/whisper/commit/c8b1d43)

This "pip-installable by default" convention makes OpenAI's code immediately usable without cloning or path manipulation — a cross-cutting pattern that unifies the otherwise divergent SDK and research styles. See [[dependencies]] for build tool choices.
