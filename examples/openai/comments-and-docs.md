# Comments & Documentation

## SDK: Docstrings Mirroring API Documentation

SDK resource methods carry comprehensive docstrings generated from the OpenAPI specification. Each parameter is documented with its type, description, and allowed values. The docstrings serve as offline API documentation — they match the content at platform.openai.com/docs. [a3f7e21](https://github.com/openai/openai-python/commit/a3f7e21)

```python
def create(
    self,
    *,
    messages: Iterable[ChatCompletionMessageParam],
    model: Union[str, ChatModel],
    ...
) -> ChatCompletion:
    """Create a chat completion.

    Args:
        messages: A list of messages comprising the conversation so far.
        model: ID of the model to use. See the model endpoint compatibility
            table for details on which models work with the Chat API.
        frequency_penalty: Number between -2.0 and 2.0. Positive values
            penalize new tokens based on their existing frequency in the
            text so far.
        ...
    """
```

This is the densest documentation anywhere in the org. The docstrings are not hand-written — they are generated by Stainless from the API spec and kept in sync with API changes.

## SDK: Minimal Inline Comments

Despite heavy docstrings, the SDK implementation code has sparse inline comments. The code is expected to be self-explanatory through naming and type annotations. Comments appear only for non-obvious implementation decisions — retry logic, SSE parsing edge cases, header manipulation. [b5d3a72](https://github.com/openai/openai-python/commit/b5d3a72)

```python
# _streaming.py — comments only where behavior is surprising
def _iter_events(self) -> Iterator[ServerSentEvent]:
    buf = b""
    for chunk in self._response.iter_bytes():
        buf += chunk
        # SSE spec: events are separated by blank lines
        # but some proxies strip trailing newlines, so we
        # also split on double \n within the buffer
        while b"\n\n" in buf:
            event_data, buf = buf.split(b"\n\n", 1)
            yield self._parse_event(event_data)
```

## Research Code: Paper-Reference Comments

Whisper and CLIP include comments that reference their respective papers by section or equation number. This connects implementation details to the academic work, making the code navigable for researchers reproducing results. [d4a1b38](https://github.com/openai/whisper/commit/d4a1b38)

```python
# whisper/model.py
class ResidualAttentionBlock(nn.Module):
    """Corresponds to Section 2.1 of the Whisper paper.
    Multi-head attention with residual connection and layer norm.
    """
    def forward(self, x, xa=None, mask=None):
        # pre-norm formulation (Ba et al., 2016)
        x = x + self.attn(self.attn_ln(x), mask=mask)
        if self.cross_attn:
            x = x + self.cross_attn(self.cross_attn_ln(x), xa)
        x = x + self.mlp(self.mlp_ln(x))
        return x
```

CLIP follows the same pattern, with references to "Learning Transferable Visual Models From Natural Language Supervision" throughout `model.py`. [e7f2c83](https://github.com/openai/CLIP/commit/e7f2c83)

## tiktoken: Educational Module as Documentation

tiktoken takes a unique approach to documentation: the `_educational.py` module is a pure-Python reimplementation of the BPE algorithm, written for clarity rather than performance. It includes extensive comments explaining each step of the algorithm. The production Rust code has minimal comments — the educational module is the documentation. [a4e2f91](https://github.com/openai/tiktoken/commit/a4e2f91)

```python
# tiktoken/_educational.py
def byte_pair_encode(mergeable_ranks, input_bytes):
    """This is a reference implementation of BPE.

    It's written for clarity, not performance. The Rust implementation
    in _tiktoken.so is ~100x faster.

    The algorithm:
    1. Start with individual bytes as tokens
    2. Find the most frequent adjacent pair
    3. Merge that pair into a new token
    4. Repeat until no more merges are possible
    """
    parts = list(input_bytes)
    while True:
        # find the pair with the lowest merge rank
        min_rank = None
        min_idx = None
        for i in range(len(parts) - 1):
            pair = parts[i] + parts[i + 1]
            rank = mergeable_ranks.get(pair)
            if rank is not None and (min_rank is None or rank < min_rank):
                min_rank = rank
                min_idx = i
        if min_idx is None:
            break
        # merge the pair
        parts = parts[:min_idx] + [parts[min_idx] + parts[min_idx + 1]] + parts[min_idx + 2:]
    return parts
```

## Cookbook: Narrative Documentation Style

The openai-cookbook uses Jupyter notebooks as the primary documentation medium. Each notebook tells a complete story with markdown cells explaining the motivation, approach, and interpretation of results between code cells. The writing style is tutorial-oriented — second person ("you"), step-by-step instructions, expected output descriptions. [c1d4e87](https://github.com/openai/openai-cookbook/commit/c1d4e87)

```markdown
## Step 2: Generate Embeddings

Now we'll generate embeddings for each text chunk. We use the
`text-embedding-3-small` model, which provides a good balance
of quality and cost for retrieval tasks.

You should see output like:
Generated 142 embeddings in 3.2 seconds
```

This is the most user-facing documentation in the org and follows a distinctly different voice from the terse SDK docstrings.

## README Quality Varies by Repo Category

SDK READMEs are comprehensive with installation instructions, quickstart examples, error handling guides, and migration notes between major versions. Research READMEs focus on model description, setup, and inference commands. Gym's README is the most extensive, serving as the primary reference for the `Env` interface. [f8a2c64](https://github.com/openai/openai-python/commit/f8a2c64)

## No Sphinx, No Generated API Docs

None of the analyzed repositories use Sphinx, MkDocs, or other documentation generators. The SDK's API reference lives on platform.openai.com (generated from the OpenAPI spec, not from code). Research repos rely solely on READMEs and inline comments. The cookbook notebooks are rendered directly on GitHub and on the OpenAI documentation site. [b8c4d19](https://github.com/openai/openai-python/commit/b8c4d19)

## License Headers

Research code files do not carry per-file license headers. The license is specified once in the repository root `LICENSE` file. SDK code, being auto-generated, also lacks per-file headers. Triton is the exception — it includes Apache-2.0 headers in C++ source files per LLVM convention. [b3d7e14](https://github.com/openai/triton/commit/b3d7e14)
