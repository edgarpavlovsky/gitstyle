# Testing

## SDK: Generated Test Suites (Stainless)

The SDKs ship extensive tests largely auto-generated by Stainless. Tests cover every resource method, verifying request serialization, response deserialization, and error handling. A notable convention: tests validate _type shapes_ rather than _content values_. The `assert_matches_type` helper checks that responses conform to Pydantic models, not that they contain specific text — making tests deterministic without mocking. [a3f7e21](https://github.com/openai/openai-python/commit/a3f7e21) [b8c4d19](https://github.com/openai/openai-python/commit/b8c4d19)

```python
class TestCompletions:
    parametrize = pytest.mark.parametrize("client", [sync_client, async_client], indirect=True)

    @parametrize
    def test_create(self, client: OpenAI) -> None:
        completion = client.chat.completions.create(
            messages=[{"role": "user", "content": "Say this is a test"}],
            model="gpt-4",
        )
        assert_matches_type(ChatCompletion, completion, path=["response"])
```

## SDK: Mock Server over HTTP Mocks

Rather than mocking HTTP calls, SDK tests use a lightweight mock server (Prism, from Stainless) that serves responses matching the OpenAPI spec. This catches serialization bugs that unit-level mocks would miss — the mock server enforces the actual API contract. [c2e9a47](https://github.com/openai/openai-python/commit/c2e9a47)

## SDK: Parametrized Sync/Async Parity

Both sync and async clients run through the same test logic via pytest parametrize. Each test executes twice, ensuring behavioral parity between interfaces. [d2f1a93](https://github.com/openai/openai-python/commit/d2f1a93)

## Research Code: Minimal Formal Testing

Research repos have thin or absent test suites. Whisper has no `tests/` directory; CLIP likewise. Correctness is validated by running inference on known inputs and checking against published benchmarks. The implicit contract: "if the model reproduces published metrics, the implementation is correct." This is standard for ML research code where the output is a trained model, not an API. [d4a1b38](https://github.com/openai/whisper/commit/d4a1b38)

## gym: Testing as Interface Enforcement

Gym has a unique pattern: `gym.utils.env_checker.check_env()` validates that _any_ environment conforming to the `Env` interface returns correct tuple shapes, has valid spaces, and produces valid initial observations. Third-party environments use `check_env` to verify compatibility. This is testing as contract enforcement rather than implementation validation. [f1a9d52](https://github.com/openai/gym/commit/f1a9d52)

```python
from gym.utils.env_checker import check_env
env = gym.make("CartPole-v1")
check_env(env)  # validates interface compliance
```

## tiktoken: Round-Trip Correctness and Performance Regression

tiktoken tests focus on the encode-then-decode invariant: the round trip should produce the original text. The suite covers Unicode edge cases, special tokens, and BPE merge boundaries. Critically, tests also benchmark encoding performance to catch regressions in the Rust core — a 10% slowdown would indicate a problem in the BPE merge algorithm. This dual focus on correctness and performance is unique to tiktoken within the org. [a4e2f91](https://github.com/openai/tiktoken/commit/a4e2f91)

```python
def test_roundtrip():
    enc = tiktoken.get_encoding("cl100k_base")
    for text in SAMPLE_TEXTS:
        assert enc.decode(enc.encode(text)) == text

def test_special_tokens():
    enc = tiktoken.get_encoding("cl100k_base")
    tokens = enc.encode("<|endoftext|>", allowed_special={"<|endoftext|>"})
    assert tokens == [100257]
```

The `_educational.py` module also serves as an implicit test oracle — its pure-Python output can be compared against the Rust implementation to verify correctness without relying on expected-value fixtures.

## Triton: LLVM-Style FileCheck Tests

Triton adopts LLVM/MLIR testing conventions — the only repo in the org using FileCheck-style assertions against compiler IR output. Tests invoke JIT-compiled GPU kernels and verify numerical correctness against PyTorch reference implementations. [b3d7e14](https://github.com/openai/triton/commit/b3d7e14)

```python
@triton.jit
def kernel(x_ptr, output_ptr, n_elements, BLOCK_SIZE: tl.constexpr):
    offsets = tl.arange(0, BLOCK_SIZE)
    x = tl.load(x_ptr + offsets)
    tl.store(output_ptr + offsets, x * 2)

def test_double():
    x = torch.randn(1024, device='cuda')
    output = torch.empty_like(x)
    kernel[(1,)](x, output, 1024, BLOCK_SIZE=1024)
    torch.testing.assert_close(output, x * 2)
```

## CI and Coverage

OpenAI standardizes on GitHub Actions. SDK repos test every PR against the mock server. tiktoken CI builds the Rust extension and tests on Linux, macOS, and Windows. Research repos have lighter CI, often just linting. No repository enforces coverage thresholds — SDK tests are comprehensive by construction, research repos do not measure coverage at all. [e8b3c19](https://github.com/openai/openai-python/commit/e8b3c19)
