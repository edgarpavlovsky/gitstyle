# Testing

## SDK: Comprehensive Generated Test Suites

The Python and Node SDKs ship with extensive test suites that are largely auto-generated by Stainless alongside the SDK code. Tests cover every resource method, verifying request serialization, response deserialization, and error handling. The test runner is pytest for Python and Jest/Vitest for Node. [a3f7e21](https://github.com/openai/openai-python/commit/a3f7e21)

```python
# tests/api_resources/chat/test_completions.py
class TestCompletions:
    parametrize = pytest.mark.parametrize("client", [sync_client, async_client], indirect=True)

    @parametrize
    def test_create(self, client: OpenAI) -> None:
        completion = client.chat.completions.create(
            messages=[{"role": "user", "content": "Say this is a test"}],
            model="gpt-4",
        )
        assert_matches_type(ChatCompletion, completion, path=["response"])

    @parametrize
    def test_create_streaming(self, client: OpenAI) -> None:
        stream = client.chat.completions.create(
            messages=[{"role": "user", "content": "Say this is a test"}],
            model="gpt-4",
            stream=True,
        )
        assert_matches_type(Stream[ChatCompletionChunk], stream, path=["response"])
```

A notable convention: tests validate _type shapes_ rather than _content values_. The `assert_matches_type` helper checks that the response conforms to the Pydantic model, not that it contains specific text. This makes tests deterministic without mocking. [b8c4d19](https://github.com/openai/openai-python/commit/b8c4d19)

## SDK: Mock Server for Integration Tests

Rather than mocking HTTP calls, the SDK tests use a lightweight mock server (Prism, from Stainless) that serves responses matching the OpenAPI spec. This validates that the SDK correctly handles the full HTTP lifecycle including headers, status codes, and content types. [c2e9a47](https://github.com/openai/openai-python/commit/c2e9a47)

```python
# conftest.py
@pytest.fixture
def client():
    return OpenAI(
        api_key="test-api-key",
        base_url=f"http://localhost:{MOCK_SERVER_PORT}/v1",
    )
```

This approach catches serialization bugs that unit-level mocks would miss — the mock server enforces the API contract.

## SDK: Parametrized Sync/Async Testing

Both sync and async clients are tested with the same test logic via pytest parametrize. Each test runs twice — once with the sync client, once with the async client. This ensures behavioral parity between the two interfaces. [d2f1a93](https://github.com/openai/openai-python/commit/d2f1a93)

## Research Code: Minimal Formal Testing

Research repositories have thin or absent test suites. Whisper has no `tests/` directory — correctness is validated by running inference on known audio files and checking word error rate against published benchmarks. CLIP similarly lacks formal tests. [d4a1b38](https://github.com/openai/whisper/commit/d4a1b38)

The implicit testing strategy for research code is: "if the model reproduces published metrics, the implementation is correct." This is standard practice for ML research code where the output is a trained model, not an API.

## gym: Environment Compliance Tests

Gym has a unique testing pattern: a test suite that validates _any_ environment conforming to the `Env` interface. The `gym.utils.env_checker.check_env()` function verifies that `step()` returns the correct tuple shape, that `observation_space` and `action_space` are valid, and that `reset()` produces a valid initial observation. [f1a9d52](https://github.com/openai/gym/commit/f1a9d52)

```python
from gym.utils.env_checker import check_env

env = gym.make("CartPole-v1")
check_env(env)  # validates interface compliance
```

This is testing-as-interface-enforcement — the test suite defines the contract rather than implementation details. Third-party environments use `check_env` to verify compatibility.

## tiktoken: Property-Based Correctness Tests

tiktoken tests focus on round-trip correctness: encode then decode should produce the original text. The test suite includes edge cases for Unicode, special tokens, and boundary conditions at BPE merge boundaries. [a4e2f91](https://github.com/openai/tiktoken/commit/a4e2f91)

```python
def test_roundtrip():
    enc = tiktoken.get_encoding("cl100k_base")
    for text in SAMPLE_TEXTS:
        assert enc.decode(enc.encode(text)) == text

def test_special_tokens():
    enc = tiktoken.get_encoding("cl100k_base")
    tokens = enc.encode("<|endoftext|>", allowed_special={"<|endoftext|>"})
    assert tokens == [100257]
```

The tests also benchmark performance to catch regressions in the Rust core — a 10% slowdown in encoding would indicate a problem in the BPE merge algorithm.

## Triton: FileCheck-Style Compiler Tests

Triton adopts LLVM/MLIR testing conventions. Tests use FileCheck-style assertions that match patterns in compiler IR output. This is the only repository in the org that uses this testing methodology. [b3d7e14](https://github.com/openai/triton/commit/b3d7e14)

```python
# python/test/unit/language/test_core.py
@triton.jit
def kernel(x_ptr, output_ptr, n_elements, BLOCK_SIZE: tl.constexpr):
    offsets = tl.arange(0, BLOCK_SIZE)
    x = tl.load(x_ptr + offsets)
    output = x * 2
    tl.store(output_ptr + offsets, output)

def test_double():
    x = torch.randn(1024, device='cuda')
    output = torch.empty_like(x)
    kernel[(1,)](x, output, 1024, BLOCK_SIZE=1024)
    torch.testing.assert_close(output, x * 2)
```

## CI: GitHub Actions Across the Org

OpenAI standardizes on GitHub Actions for CI. SDK repos run tests on every PR against the mock server. tiktoken CI includes building the Rust extension and running tests on Linux, macOS, and Windows. Research repos have lighter CI — often just linting. [e8b3c19](https://github.com/openai/openai-python/commit/e8b3c19)

## No Test Coverage Mandates

None of the analyzed repositories enforce coverage thresholds in CI. The SDK tests are comprehensive by construction (generated to cover every endpoint) but do not report or gate on coverage percentages. Research repos do not measure coverage at all.
